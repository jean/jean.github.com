Plone Development Layouts

Setting up a Plone development environment is quite a complicated business. For example:

  • You need to have the right Python version,
  • with the right level of isolation from the system Python
  • as well as from other locally compiled Pythons.
  • You need suitably compiled libxml2, libjpeg, zlib, etc.

There have been various attempts to address this, none yet revealed as the Right Way. Ã‰rico Andrei nails it:

Last night while talking to Clayton and Liz about the whole ZopeSkel/Paster/Templer/Crushinator situation it became clear that the story for new developers is a complete mess.
There are good ideas, great developers and high expectations. There isn't a working solution. Some people jumped on Templer, most people still need to pin ZopeSkel <2.99 (damn local commands).
This is bad, and having a new shiny thing that "eases" Plone development just made the situation even worse -- as long as someone must use it and then <zopeskel create> & <paster addcontent>.
Time to kill most of the available options, focus on just one and have a compelling story to tell to newcomers.

Add-on template

Author
Mikko Ohtamaa
Source
https://github.com/miohtama/sane_plone_addon_template
Supports
Plone 4.1, ...?

Based on Plone 4.1, five.grok. It is not based on any template engine: just clone and go. Tries to be as minimal as possible.

Martin Aspeli asks: "Did you have a look at the 'plone-customizations' egg provided by plone-devstart? Wondering if it serves a somewhat similar purpose."

Cons

  • It's opinionated, e.g. choosing CMF skin layers instead of z3c.jbot and views.

Plone Devstart

Author
Martin Aspeli
Source
https://github.com/plone/plone-devstart
Supports
Plone 4.1.*, ...?

"plone-devstart.py is a script to quickly and safely bootstrap a useful Plone development environment."

At its simplest, this is similar to Alex's Hello Plone, except automated and dynamic. It tries to check whether a sane environment exists.

More importantly, it can be updated as best practices evolve.

Pros

  • Tries to encode best practices as running code.
  • Supports various Plone versions.
  • Tries to accommodate your local setup (uses existing Python if suitable).
  • Requires only the standard Python library.

Cons

  • Checks using heuristics. A Python that looks OK may still surprise you.
  • Reinvents autoconf/automake (but those are still complicated and not cross-platform).

ZopeSkel

Author
Cris Ewing
Source
http://pypi.python.org/pypi/ZopeSkel
Supports
Plone 3.*,
Plone 4.*
...?

Used to be based on paster, now moving to templer. From a quick look it appears that the latest (3.0a1) isn't useful yet, so install with version specified as <= 2.99.

Cons

  • It has accumulated quite a few crufty templates that are incomplete or haven't been updated to current development practices.
  • It can be hard to figure out what exactly a template does or which is the right one to use for some purpose.
  • There are a lot of obsolete tutorials floating around. At the least there should be a ZopeSkel site which lists obsolete tutorials: a "don't read these" list (quick wiki page).
  • It's unfinished:
Our problem with ZopeSkel goes beyond keeping docs in sync. We never finished ZopeSkel < 2.99 -- as it has a zopeskel script that did not support local commands -- and we decided to go with a new idea (templer) with ZopeSkel 3.0 and that is in a zombie state (again, no local commands), and then we saw this whole ZopeSkel situation deserved a new approach (Crushinator) and that is also stalled somewhere.

pip install

Support: "pip install -r 4.2.x.txt Plone" ASAP. Failures with this process become "general Python failures" not "Plone specific failures".

See http://lists.plone.org/pipermail/plone-product-developers/2012-March/011335.html

Cons

  • Doesn't actually exist.
